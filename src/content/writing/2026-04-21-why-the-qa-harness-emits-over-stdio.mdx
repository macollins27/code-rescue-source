The user had asked for the real-time findings channel to be implemented as an MCP server. My implementation plan downgraded that to <em>"MCP server or HTTP endpoint — chose HTTP for simplicity"</em> and shipped HTTP. The HTTP server lived at `scripts/qa-emit-server.ts` and exposed a single endpoint the QA agent's `browser_evaluate` would `POST` to whenever it observed a finding.

It worked in unit tests. It worked in the synthetic integration tests. It did not work the first time a real Phase 2 QA dispatch ran against the app.

**What broke.** The app's Content Security Policy `connect-src` directive rejected `browser_evaluate` fetches from `localhost:3011` to `127.0.0.1:3012`. The agent's emit calls came back with CSP errors. The agent — correctly — fell back to its secondary path, which was to accumulate findings in memory and emit them in a single `Write` at the end of the discovery run. Then the runner's 60-minute wall-clock watchdog fired and killed the subprocess, because discovery had taken longer than the budget. The batch `Write` never executed.

**What was lost.** A 67-tool-call discovery run. The agent had already root-caused a CRITICAL `o.map` bug and reproduced five reference findings from the March-29 baseline. None of it survived to disk. The agent had done correct, valuable work. The emission layer ate all of it.

**The mandate was unambiguous: no further QA dispatches until the emission channel was rebuilt, and the rebuild had to ship as one body of work — no deferring it across sessions.** The fix had to be architectural, not a patch.

**The fix is the title of the post.** The new emission layer is a stdio MCP server at `scripts/qa-emit-mcp.ts`. The transport is JSON-RPC over the `claude -p` subprocess's stdin and stdout. Bytes move through process pipes. There is no network socket. There is no `fetch`. There is no browser context to enforce CSP against. The thing that killed the prior implementation can no longer kill this one because the failure mode requires a policy that does not apply to this transport.

Stdio MCP also forces the durability primitive earlier in the path. The shared writer at `scripts/qa-emit-writer.ts` does an `O_EXCL` lockfile, an `fsync` per write, and TTL-based stale-holder recovery. It is imported by both the MCP server and the in-process deterministic probes (`qa-prescan`, `qa-fuzz-inputs`, `qa-state-walk`, `qa-contract-check`). One write path, one lock convention, one recovery story. The HTTP server had its own ad-hoc append logic — three slightly-different implementations across the harness.

**The 12-test gate.** Before any new Phase 2 dispatch runs, twelve runnable tests exercise the emission chain end-to-end. T1 through T4 hit the writer directly: a basic three-append, a 10-process × 100-write × 8-KB-payload cross-process atomicity test (above PIPE_BUF), a stale-lock reclaim, and a SIGKILL-during-burst durability check. T5 through T10 spawn the MCP server and drive it via the official SDK client: handshake, tools/list, full lifecycle, invalid-input survival, SIGTERM drain, SIGKILL during a 20-emit burst. T11 and T12 are the integration test that matters most — a real `claude -p` subprocess against a runner-shape `.mcp-test.json` configuration with `--strict-mcp-config` and `--allowedTools mcp__qa_emit__finding_emit`, prompted to call the tool exactly once. T12 is the last gate before any real QA dispatch. It proves the runner → config → `claude -p` → stdio → MCP → writer → disk chain without spending budget on a real flow.

**The lesson generalizes.** When an agent's output channel can be eaten by a downstream policy the agent cannot see, the channel is wrong. Browser CSP is exactly that kind of policy: invisible from the agent's vantage, enforced silently, and the failure mode is "the data you wrote disappears." Process stdio is the opposite — there is no policy layer between the writing process and the receiving process other than the operating system's pipe semantics. For agent emission specifically, stdio is the safe transport. For anything else under the control of a browser context, the question to ask first is: what policy could refuse the transport, and is the agent positioned to know?
